Bean, DTO, Entity의 구분

1. Bean

  • 역할: 사용자의 입력 데이터를 저장하고, 간단한 검증(if문 등)을 포함할 수 있는 객체
  • 사용 시점: 입력 폼이나 사용자 요청 데이터를 받을 때
  • 특징:
    • 단순한 형태의 자바 클래스(getter/setter 포함)
    • 필요한 경우 코드 내에서 직접 유효성 검증 수행
    • 화면(Form)과 밀접한 관계를 가짐
  • 예시: 
    public class UserBean {
    private String name;
    private String email;

    public String getName() { return name; }
    public void setName(String name) {
        if(name == null || name.isEmpty()) {
            throw new IllegalArgumentException("이름은 필수 입력값입니다.");
        }
        this.name = name;
    }

    public String getEmail() { return email; }
    public void setEmail(String email) { this.email = email; }
}

2. DTO (Data Transfer Object)

  • 역할: 계층 간(Controller ↔ Service ↔ DAO) 데이터를 전달하는 전용 객체
  • 특징:
    • 검증 로직 없이 순수 데이터 전달 목적
    • 직렬화나 변환(JSON, XML) 시 주로 사용
    • 일반적으로 getter/setter만 포함
  • 예시: 
    public class UserDto {
    private int id;
    private String name;
    private String email;

    public int getId() { return id; }
    public void setId(int id) { this.id = id; }

    public String getName() { return name; }
    public void setName(String name) { this.name = name; }

    public String getEmail() { return email; }
    public void setEmail(String email) { this.email = email; }
}

3. Entity

  • 역할: 데이터베이스의 테이블 구조와 직접 연결되는 객체
  • 특징:
    • 테이블 컬럼과 동일한 필드를 가짐
    • DB에서 조회한 데이터를 담거나 저장할 때 사용
    • JDBC 단계에서는 SQL 결과를 담는 단순 데이터 구조
  • 예시: 
    public class UserEntity {
    private int id;
    private String name;
    private String email;
    private String createdAt;

    // Getter/Setter 생략
}

4. 서비스 전용 DTO

  • 필요 시점: DB 데이터(Entity) 외에 비즈니스 로직에서 가공된 값을 전달해야 할 때
  • 역할: 여러 테이블을 조합하거나, 계산된 결과를 담는 전용 DTO
  • 특징:
    • DB 구조와 다를 수 있음
    • Entity를 변환하거나, 추가 연산 결과를 포함
    • Service 계층 전용으로 설계
  • 예시: 
    public class UserSummaryDto {
    private String name;
    private int postCount;
    private double avgLike;

    // Getter/Setter 생략
}

정리

  • Bean: 입력값 저장 및 수동 검증용
  • DTO: 계층 간 데이터 전달용
  • Entity: DB 데이터 저장 및 조회용
  • 서비스 전용 DTO: 비즈니스 로직 가공 결과 전달용

→ JDBC 단계에서는 @Valid와 같은 자동 검증 어노테이션이 동작하지 않으므로, Bean에서는 간단한 if문 검증 방식을 사용합니다.
